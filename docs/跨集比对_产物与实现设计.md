# 跨集比对：产物形态 + 文章模板 + Prompt 框架

> 这是产品的招牌能力。核心原则一句话：**先把每期拆成带来源的"原子观点"，再围绕子议题聚类，最后才生成矩阵和文章——绝不让模型"分别总结再拼一起"。**

---

## 0. 为什么不能一步到位

如果直接把 5 期内容塞给模型说"做个交叉比对"，它会偷懒：分别总结每期，再写一段"综上所述"。这就是伪比对。

真比对必须**强制结构化中间产物**，把"比"这个动作拆出来：

```
阶段 A：观点抽取   每期 → 一组原子观点（claim），每条带来源徽标
        ↓
阶段 B：议题聚类   把所有 claim 按"在讲同一件事"归并成子议题
        ↓
阶段 C：立场对齐   每个子议题下，对齐各主播的立场，判定 共识/分歧/独有/存疑
        ↓
阶段 D：成稿       基于 C 的结构化结果生成 观点矩阵 + 专题文章
```

每一阶段输出 JSON，下一阶段吃上一阶段的 JSON。这样"比对"是数据操作，不是模型的自由发挥，可追溯、可标来源、可验证。

## 1. 中间数据结构

### 1.1 原子观点 claim（阶段 A 产出）

```json
{
  "claim_id": "ep3_c07",
  "episode_id": "ep3",
  "speaker": "主播/嘉宾名，未知则 unknown",
  "text": "观点的一句话陈述（模型归纳，不是逐字稿）",
  "quote": "支撑这条观点的原话片段（仅当 source=asr/official/user_paste 时有）",
  "topic_hint": "模型初判它属于哪个子议题",
  "stance": "断言型 | 预测型 | 价值判断 | 经验分享 | 反驳他人",
  "source": "asr | official | user_paste | shownotes | web | inferred",
  "confidence": "high | medium | low"
}
```

要点：`source` 和 `quote` 是后面可信度护栏的命根子。只有听稿类来源才有 `quote`，简介/推断来源没有原话、`confidence` 自动降级。

### 1.2 子议题 subtopic（阶段 B/C 产出）

```json
{
  "subtopic_id": "st_02",
  "title": "子议题标题，如『AI 会不会取代初级程序员』",
  "positions": [
    {
      "episode_id": "ep1",
      "speaker": "...",
      "summary": "该期在这个子议题上的立场（一两句）",
      "claim_ids": ["ep1_c03", "ep1_c11"],
      "evidence_strength": "full | segment | shownotes_only | inferred"
    }
  ],
  "relation": "共识 | 分歧 | 部分分歧 | 单方独有 | 信息冲突",
  "divergence_root": "若分歧，根源是什么：立场不同 / 信息不同 / 时间点不同 / 定义不同 / 无法判断",
  "note": "需要提醒读者的存疑点"
}
```

## 2. 招牌产物：观点矩阵

文章顶部最显眼的位置放这张表。它是"这工具不是摘要器"的一眼证明。

```
┌─────────────────────┬──────────┬──────────┬──────────┬─────────┐
│ 子议题 \ 来源        │ 主播A    │ 嘉宾B    │ 主播C    │ 关系    │
├─────────────────────┼──────────┼──────────┼──────────┼─────────┤
│ AI取代初级程序员      │ 会，2年内 │ 不会,只是 │ 部分会,   │ ⚠分歧   │
│                     │ 🎧       │ 工具变化🎧│ 看领域📝  │         │
├─────────────────────┼──────────┼──────────┼──────────┼─────────┤
│ 该不该现在学AI编程    │ 必须学🎧  │ 必须学🎧  │ 必须学🎧  │ ✅共识  │
├─────────────────────┼──────────┼──────────┼──────────┼─────────┤
│ 提示工程会消失吗      │ —        │ 会消失🎧  │ —        │ 💡独有  │
└─────────────────────┴──────────┴──────────┴──────────┴─────────┘

徽标：🎧听稿 📝简介 🌐联网 🤔推断    关系：✅共识 ⚠分歧 💡独有 ❓信息冲突
```

交互（HTML 里）：点格子 → 弹出该立场对应的 `claim` 和原话 `quote`；分歧行可展开看 `divergence_root`。

矩阵设计的三条铁律：

1. **分歧行优先排在上面**——分歧才是读者最想看的，共识可以折叠。
2. **每个立场必带来源徽标**——一眼看出哪条是真听到的、哪条是推断的。
3. **空格子如实留空（—）**——某主播没谈这个议题就留空，绝不为了填满而编。

## 3. 专题文章章节模板

```markdown
# {主题}：来自 {N} 期播客的交叉解读

> 本专题对比了 {节目列表}，共 {N} 期。
> 证据构成：🎧 {x} 期全文/片段听稿 · 📝 {y} 期仅简介 · 🌐 联网补充
> 可信度：{高/中/低} —— {一句话说明，如"核心结论基于听稿，少数推断已标注"}

## 一图看懂：观点矩阵
{观点矩阵表}

## 大家吵的是什么：核心分歧
（逐个分歧子议题展开。每个分歧必须回答：谁和谁的分歧、各自怎么说、
 分歧根源是立场/信息/时间/定义不同，并带原话佐证）

## 没争议的部分：共识
（被多方反复确认的事实和判断，这部分可信度最高，适合直接采信）

## 只有 TA 说了：独到观点
（只出现在某一期的视角或反常识观点，注明出处，标"单一来源,供参考"）

## 这个话题是怎么演变的（可选）
（若多期时间跨度大，按时间线呈现讨论的变化）

## 需要存疑的地方
（信息冲突、仅凭简介推断、单一来源未经证实的点，集中提醒）

## 附：各期速览
（每期一段，含节目名、单集标题、链接、本期在该主题上的主要贡献）
```

注意章节顺序：**分歧在前、共识在后、存疑收尾**。这跟普通摘要"按集罗列"完全不同，强迫内容围绕"观点关系"组织，这就是差异化的体感来源。

## 4. Prompt 框架（四段）

### 阶段 A · 观点抽取（每期单独跑一次）

```
你在为一篇多播客交叉比对专题做素材抽取。下面是单期播客的内容，
来源类型为：{source}（asr全文 / segment片段 / shownotes简介 / user_paste）。

抽取这一期中与主题「{主题}」相关的"原子观点"。要求：
- 每条是一个独立、可与他人比较的判断或主张，不要笼统概括。
- 区分谁说的（speaker），不确定就标 unknown。
- 只有当来源是听稿类(asr/official/user_paste)时，才填 quote 原话；
  来源是简介时 quote 留空，confidence 不得高于 medium。
- 绝对不要把你的推断写成主播的原话。无法从材料判断的，不要编。
- 给每条初判它属于哪个子议题(topic_hint)。

输出 JSON 数组，schema：{claim 结构}
```

### 阶段 B · 议题聚类（吃所有期的 claim）

```
下面是来自 {N} 期播客的全部原子观点(claims)。请把它们归并成若干"子议题"——
即"在讨论同一件事"的观点应归到一组，哪怕用词不同、立场相反。

要求：
- 子议题要足够具体，能让不同主播在其下形成可比较的立场。
- 一条 claim 可归入最相关的一个子议题。
- 不要遗漏只有一期提到的观点，它们可能是"独有观点"。

输出子议题列表，每个含 title 和归入的 claim_ids。
```

### 阶段 C · 立场对齐与关系判定（逐子议题）

```
针对子议题「{title}」，下面是各期在此议题下的 claims（含来源与原话）。

对齐各主播立场并判定关系：
- 给每期一句话 summary，标 evidence_strength（full/segment/shownotes_only/inferred）。
- 判定整体 relation：共识 / 分歧 / 部分分歧 / 单方独有 / 信息冲突。
- 若分歧，判断 divergence_root：立场不同 / 信息不同 / 时间点不同 / 定义不同 / 无法判断。
- 重要：如果某期立场是基于 shownotes_only 或 inferred 得出的，
  在 note 里明确提醒"此立场为简介推断，未经听稿确认"。
- 不要为了制造对立而夸大分歧，也不要为了和谐而抹平真实分歧。

输出 subtopic 结构。
```

### 阶段 D · 成稿

```
基于以下结构化比对结果(subtopics)，按给定模板生成专题文章和观点矩阵。

铁律：
- 每个立场陈述后必须带来源徽标。
- 分歧章节在前，共识在后，存疑收尾。
- 只能用 source 为 asr/official/user_paste 的句子做"可分享金句"候选，并单独列出。
- 凡 evidence_strength 为 shownotes_only/inferred 的内容，措辞用"据简介"/"推测"，
  不得写成"主播明确表示"。
- 不在材料里的事实不要补充；联网补充的内容单独标 🌐 并说明用途。
```

## 5. 这套设计如何兜住可信度

把第一份方案的 Q9 落到了实处，三道闸：

1. **抽取阶段**：来源决定 quote 有无、confidence 上限。简介推断进不了高可信。
2. **对齐阶段**：每个立场带 evidence_strength，简介推断的分歧会被打上"未经听稿确认"。
3. **成稿阶段**：措辞分级 + 金句只许用真实听稿。

结果：一篇专题里，读者能清楚区分"三个主播确实当面吵过这件事"和"我们从两份简介里推测他们可能不一致"。后者也有价值（提示去深入），但绝不冒充前者。

---

## 6. 这也回答了"先快后全"的张力

跨集比对天然分两层质量：

- **v1（简介聚类）**：能出矩阵骨架，但分歧多半标着"简介推断"。适合让用户先看到结构、决定值不值得深挖。
- **v2（关键段/全文听稿）**：分歧有原话佐证，evidence_strength 升级，金句解锁。

所以默认策略微调：**只要用户做的是"主题专题"（多期），就把额度优先花在"各期与该主题最相关的段落"上**——这正是片段转写最该用的地方。把好钢用在跨集这把刀的刀刃上。

## 7. 交给 codex 的落地清单

1. 定义 `claim` / `subtopic` 两个数据结构（第 1 节），落库或落 JSON 文件。
2. 实现阶段 A-D 四段流水线，每段独立可调试、可缓存中间结果。
3. claim 抽取结果进 transcript 缓存的衍生层（按 episode_id + 主题缓存，换主题可复用 claim）。
4. HTML 渲染观点矩阵组件（可点开看原话、分歧可展开）。
5. 文章按第 3 节模板渲染，来源徽标贯穿全程。
6. 金句图生成器加校验：只接受 source∈{asr,official,user_paste} 的句子。
