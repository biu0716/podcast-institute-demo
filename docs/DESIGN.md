# 设计文档 · 小宇宙播客专题研究助手（个人项目版）

> 定位：**自己用 + GitHub 作品集展示**。
> 因此本文档已砍掉所有 SaaS 包袱（多人队列、配额、共享缓存、付费逻辑、片段转写、截图 OCR），
> 只保留自用所需 + 值得展示的核心能力。

---

## 1. 一句话定位（北极星，不可动摇）

> **把一个你关心的问题，变成一篇可阅读、可分享的播客专题。**
> 输入是「问题」而非「单集」；输出是跨多期回答这个问题的专题 + 观点矩阵 + 来源可信度标注。

**与转写工具的品类区别**：转写工具帮你"把听过的存下来"（输入=一集，输出=文字）；
本产品帮你"把没空听的搞懂"（输入=一个问题，输出=跨集专题）。不在速度/成本上跟转写工具卷——
那是它们的主场；本产品的主场是**选题、互补、跨集比对、可信度标注**，这些转写工具不做。

### ⭐ 最重要的收束判断
> **产品价值在「问题 → 专题 → 跨集比对 → 可信标注」这条链上。全文转写只是其中一格的"增强项"，不是核心路径。**
> 验证：测试专题《AI 下一步》3 期里只有 1 期有听稿、另 2 期靠简介，仍产出了有跨集比对且诚实标注来源的专题。
> 过去的痛苦都源于把"增强项"当"核心路径"死磕。把它放回增强项的位置，速度/成本/限流的矛盾同时松开。

## 2. 设计原则（也是展示亮点）

1. **从问题出发，不从单集出发**：核心是把问题变专题，转写是可选增强。
2. **本地优先、隐私友好**：数据都在本机，用户用自己的 API key，不上传第三方。
3. **多来源 + 自动降级**：能不花钱就不花钱，要花也极少，永远有免费兜底。
4. **可信度可追溯**：每个结论标来源（听稿/简介/核验/推断），绝不把 AI 推断写成节目事实。
5. **速读先交付，精读后台补**：速读版秒级给观点地图；精读版默认后台化 + 完成通知，绝不让用户盯进度条。
6. **合规**：只用合法来源（用户自己的音频转写 / 公开视频字幕 / 用户主动粘贴），不逆向、不自动化抓取平台内容。

### 术语（统一用这套，已弃用"骨架/证据"）
- **速读版**：基于公开资料、简介、show notes、已有缓存，秒级交付的观点地图。
- **精读版**：补入全文依据（ASR 或用户粘贴文稿）后的增强版，后台生成。
- 来源四档（可信度从硬到软）：**听稿 › 简介 › 核验 › 推断**。

## 3. 全文获取策略（成本/摩擦从低到高，命中即停）

**核心变化：默认不再"全文研究所有期"。** 即时全文只用在**用户主动标星的那 1 期**上，其余靠简介 + 后台慢补。
这样一次只转 1 期、不是 3 期 4 小时，基本撞不上 Groq 限流。

```
1. transcript 缓存命中            0 成本 0 操作   —— 同一集永不重复处理
2. 用户粘贴文稿（标星那期）        0 成本 1 个动作 —— 针对用户最在乎的 1 期，即时升级
3. 视频平台字幕 (yt-dlp)          0 成本 0 操作   —— 节目同时在 YouTube/B站时
4. Groq 转写（仅标星 1 期）        ~$0.04，几秒    —— 一次只转 1 期，避开限流
5. 本地 faster-whisper 后台       0 成本，慢      —— 其余期次的免费兜底，不催不倒计时
```

**关于限流**（这次踩的坑）：Groq on-demand 约 7200 秒/小时，3 期长播客必然撞上。
解法不是"等额度恢复"，而是**默认只即时转 1 期**，把"全文研究所有期"从默认行为里去掉。

**关于成本**：Groq Whisper Turbo 约 $0.04/小时且有每日免费额度，个人自用大概率全免费；
本地 Whisper 是 0 成本兜底。这是个"一年几十块甚至全免费"的个人工具。

> **已放弃且不实现**：小宇宙文稿因不让复制纯文本（只能选中生成图片），其滚动截图 OCR 抓取方案——体验差、不稳定、且在开源项目里有合规风险。直接用廉价转写替代。

## 4. 默认流水线（问题 → 速读版 → 可选精读版）

```
[输入：问题 or 链接]
  │
  ▼ 【速读版 · 秒级~30秒，必然交付，不启动完整代理】
  1. 检索候选小宇宙节目（API 调用，确定性）
  2. 按规则选片（近期 / 高影响力 / 观点互补）
  3. 各期查 transcript 缓存；其余抓公开页面（简介/show notes/时间轴）
  4. ⭐ 用简介+缓存秒级生成「观点地图」（矩阵 + 节目卡片），来源标"简介依据"
  →  立刻标完成、跳阅读页

  ▼ 【精读版 · 用户主动触发，默认后台化】
  5. 仅对用户「标星的 1 期」即时获取全文（粘贴文稿 / Groq）
  6. 完成 → 该期格子从"简介"升级为"听稿"，矩阵变扎实
  7. 其余期次本地 Whisper 后台慢补，不催不倒计时；完成发通知
```

要点见 5.1：**速读版与完整代理彻底解耦**（这是"假快"的根因）；**精读版默认后台化**；**即时全文只用在标星 1 期**。

## 5. 核心能力：跨集交叉比对（项目的招牌）

绝不"分别总结再拼一起"。强制四段结构化流水线，每段输出 JSON 喂下一段：

```
A 观点抽取   每期 → 原子观点 claim（带 speaker / quote / source / confidence）
B 议题聚类   所有 claim 按"在讲同一件事"归并成子议题
C 立场对齐   每子议题判定 共识/分歧/独有/存疑，分歧再判根源
D 成稿       生成观点矩阵 + 专题文章
```

### 关键数据结构

```jsonc
// claim（阶段 A）
{ "claim_id","episode_id","speaker","text","quote","topic_hint",
  "stance","source","confidence" }

// subtopic（阶段 B/C）
{ "subtopic_id","title",
  "positions":[{"episode_id","speaker","summary","claim_ids","evidence_strength"}],
  "relation","divergence_root","note" }
```

### 招牌产物：观点矩阵
行=子议题，列=各主播，格=立场（带来源徽标），冲突高亮，点格看原话。
铁律：分歧行排前、每个立场带徽标、没谈的议题如实留空。

### 文章章节顺序
观点矩阵 → 核心分歧 → 共识 → 独到观点 →（演变时间线）→ 存疑提醒 → 各期速览。
**分歧在前、共识在后、存疑收尾**——围绕"观点关系"组织，区别于普通摘要。

## 5.1 ⭐ 核心架构决策：速读版（观点地图）与精读版（全文增强）

> 本项目最重要的一条架构决策，直接决定核心功能能否扛住 ASR 限流。
> 起因：Groq on-demand 约 7200 秒/小时限流，3 期长播客必然撞上；且过去"速读版"底层仍启动完整
> Codex 代理去检索/筛选/写作，导致 3-5 分钟还在转——这是"假快"。

### 5.1.1 速读版 = 观点地图，且必须脱离完整代理

速读版**不是"还没写完的文章"，是"这个问题的全景结构"**。30 秒内做不出好文章，但做得出好结构。

第一屏长这样：
- 顶部一句话：**这个问题，这几期节目分别怎么答**（每期一句立场）。
- 中间一张**观点矩阵**（先用简介填，格子标"简介依据"）。
- 下面**N 张节目卡片**：节目名、单集、为什么选它、它在这个问题上的独特贡献。

**"假快"的根因与解法**：慢在"启动完整代理"。速读版只做三件**确定性、本地秒级**的事——
检索候选（API）、按规则选片、用简介跑矩阵+卡片。这三步不需要长链路 agent。
**完整代理写作只在用户点"精读版"时才启动，且后台化。** 把"写好文章"这个慢动作移出速读版关键路径。

### 5.1.2 精读版 = 全文增强，默认后台化

普通用户能盯着等的上限约 10-15 秒，精读版动辄几分钟，**必须默认后台化**：
点精读版 → 立刻回速读版继续读 → 后台补 → 完成后通知"精读版好了，点击查看"。
**等待被阅读填满 = 不焦虑。** 速读版的存在，本质就是为精读版争取后台时间。

精读不是"全文研究所有期"，而是**只对用户标星的 1 期即时取全文**（粘贴/Groq），其余后台慢补。
一次只转 1 期，基本撞不上限流。每补完一期，矩阵该期格子从"简介依据"升级为"听稿依据"，
**用户看着比对从轮廓长成有血有肉，而不是对着空白等。**

### 5.1.3 让用户愿意补文稿（针对单期、有即时反馈）

矩阵里"简介依据"的格子显示成浅色/待坐实，旁边一个小按钮：
**「📋 贴入这期文稿 → 升级为听稿依据」**。用户看到的是"贴一下这一格就变扎实"，
是针对单期、有即时回报的动作，而非"去复制一大坨"。配剪贴板监听：点"我去复制"后监听剪贴板，
检测到长中文文本自动填入、入缓存、升级该格。**永远可选**——不补也有完整速读版。

### 5.1.4 限流当常态，不当异常

转写调度器内建**额度感知队列**：入队前用 ffprobe 估时长、对比可用额度，超出的排到下一窗口并如实告知。
提交多期长播客时先算总账先告知（"3 期共 4.2 小时，超本时段额度，速读版马上给，全文分批补"），
把限流从"事后翻车"变成"事前知情"。允许标星 1 期插队。

### 5.1.5 状态模型（两层，杜绝单进度条假象）
- research 主任务：速读版就绪即标 **✅ 可读**（不等任何全文）。
- 各期转写子任务：独立状态 `done / queued / rate_limited / failed`，失败原因前台可见（如"Groq 限流"）。
- 前端：`✅ 速读版已就绪 · 🎧 精读补全中 2/3（受额度影响）`，**永远不给倒计时**。

### 5.1.6 这反而强化差异化
转写工具没有"这个结论有多可信"这个维度。本项目有，而且**用户看得见它在变强**：
矩阵每格带来源标签，顶部一句"本专题：1 期听稿坐实 · 2 期简介推断，可贴文稿升级"。
README 亮点："如何在不可靠的 ASR 限流下，让核心的跨集比对永远有交付。"

## 5.2 ⭐ 关键诊断：速读版"假快"的真正根因是架构，不是状态收尾

> 读 server.py 后确认：当前速读版（`production_mode=="quick"`）的实现，是把一整段**自然语言指令**
> （`opening + workflow + CROSS_EPISODE_COMPARISON + request`）交给 **Codex agent（subprocess.Popen）**，
> 让它自由完成"检索 → 筛选 → 写整篇文章 → 生成 HTML"。

### 为什么这必然慢
让一个**通用 agent** 去执行一个本该**确定性**的流程，它每一步都在做开放式推理 + 长文写作。
之前"检测到 HTML 就立即结束"只是**止血**——它治的是"生成完了却不收尾"，没治"生成本身就慢"。
真正的瓶颈是：**速读版不该由 agent 写文章。**

### 重构方向：速读版改成本地确定性流程，agent 只做不确定的那一步
把速读版拆开，区分"确定性步骤"（代码直接做，快）和"真正需要智能的步骤"（才调模型）：

| 步骤 | 现在 | 应改为 |
|---|---|---|
| 检索候选节目 | agent 自由检索 | 代码直接调小宇宙检索 API（确定性） |
| 选片 | agent 推理 | 代码按规则打分排序（近期/影响力/互补） |
| 抓简介/缓存 | agent 去抓 | 代码直接抓页面 + 查缓存 |
| 跨集比对（抽观点→聚类→判关系） | 混在长文里 | **唯一需要模型的一步**：一次结构化调用，输出 JSON（claim/subtopic） |
| 生成首屏 HTML | agent 写整篇文章 | **代码用 JSON 填 SVG 模板 + HTML 模板**（确定性，不写长文） |
| 写完整长文 | 速读版就写 | **移到精读版后台**，速读版不写长文 |

关键：**速读版只调一次模型（做结构化跨集比对），其余全是确定性代码。** 长文写作整体移出速读版关键路径。
这样 30 秒可达，且不依赖 Codex agent 的开放式执行。

### 5.2.1 速读版首屏 = 渐进披露（回答"字太多"）
信息架构从上到下，**默认只展开前三层，矩阵和长文折叠**：

```
1. 中心问题（小黑站在上面）
2. 小黑 SVG 放射图：3 条线 → 流程/软件/组织，图上只有这 3 个词
3. 三行「一句话回答」：每期一行立场 + 来源点(●听稿/○简介) + [补这期全文]按钮
   ——这是图与矩阵之间的桥，不点开也知道每期立场
4. ▸ 查看完整观点矩阵   （默认折叠）
5. ▸ 阅读完整文章       （默认折叠，长文是精读版后台才写的）
```

**首屏文字量标准**：用户不滚动、不点开，就能回答"这问题被拆成哪几条路、每条谁在说、哪条最值得深挖"。
渐进披露逻辑：图(秒懂结构) → 一句话(知立场) → 矩阵(看分歧) → 文章(完整论证)，在哪层停下都拿到完整价值。

### 5.2.2 小黑图：SVG 模板化，不依赖 AI PNG
- **主信息载体必须是 SVG**（文字不错、轻、稳、可自动生成）；AI PNG 只当封面装饰、不承载信息、失败回落 SVG。
- Excalidraw 手绘感要自动化 + 低文字：**预设布局模板（放射/对比/流程），手绘感用 rough.js 或固定抖动滤镜写死；
  AI 只输出少量结构化文字（中心问题 + N 个分支词 + 每支来源标记）填进模板。** 风格模板写死、内容 AI 填。

### 5.2.3 "标星一集 → 升级听稿"交互
放在首屏三行一句话里，来源点旁一个轻按钮：
`《硅谷101》Agent会先商业化落地 ○简介 [补全文]` → 点击 `补全中…`（用户继续读别的）→ 完成 `●听稿`，
该行一句话被真听稿修正、矩阵对应列同步升级。**一次只补一集、即时反馈、避开限流。**

## 6. 可信度体系（三道闸）

1. **抽取**：来源决定原话有无与 confidence 上限。简介/推断进不了高可信。
2. **对齐**：每立场带 evidence_strength；简介推断的分歧标"未经听稿确认"。
3. **成稿**：措辞分级（"主播提到" vs "据简介" vs "推测"）；金句图只许用真实听稿/字幕句子，不许用推断句。

**来源四档（可信度从硬到软，统一术语）**：**听稿 › 简介 › 核验 › 推断**
- 听稿：ASR 转写 / 用户粘贴文稿 / 视频字幕 —— 节目真说过
- 简介：节目简介 / show notes / 时间轴 —— 据公开信息推断
- 核验：联网检索补充佐证 —— 第三方资料
- 推断：AI 综合推理 —— 最软，必须明确标注

**可信度梯度要让普通用户一眼看懂**：四档配从实心到空心的小点（或深→浅颜色），
不读说明也知道"听稿最硬、推断最软"。专题顶部加一句总览：
"本专题：1 期听稿坐实 · 2 期简介推断，可贴文稿升级"。

## 7. 技术栈与模块

```
专题研究助手/
  index.html        前端：书架 / 阅读 / 书内搜索 / 选中联网检索 / 金句图 / 任务列表
  server.py         本地服务：流水线编排 / 缓存 / 转写调度 / 接口
  transcribe.py     转写：Groq(默认) + faster-whisper(本地兜底) 统一接口
  captions.py       yt-dlp 抓视频字幕 + VTT/SRT 清洗
  compare.py        跨集比对四段流水线（A→B→C→D）
  cache.db          transcript 缓存（SQLite，按 episode_id）
```

技术栈：Python + 原生 HTML/JS（无重框架，单文件前端）、SQLite、Groq API、faster-whisper、yt-dlp、ffmpeg/ffprobe（读时长）。
