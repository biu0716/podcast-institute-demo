# 小宇宙播客专题研究助手 · 整体方案

> 一句话定位：贴一个问题或播客链接，系统把多期播客整理成可读、可检索、可分享的专题；**先快后全**——2 分钟给可用初稿，再按需补全文证据。

---

## 0. 一个核心判断：不要让用户选模式

你现在列了 5 个模式（快速 / 文稿 / 直接音频 / 智能省额度 / 全文），这是**对的内部分层，错的对外呈现**。

用户的真实诉求只有一句："贴链接就出结果，别让我做决定。"

所以正确形态是：

```
对外：一个默认按钮「直接音频转写 · 推荐」
对内：一套自动降级流水线，用户感知不到模式切换
```

模式不是选择题，是系统在不同约束下自动走的分支。用户唯一需要看到的，是**结果上方一行可信度标注**（基于什么生成的），以及**需要花钱/花时间时的一次确认**。

这条判断决定了后面所有设计。

---

## 1. 默认流水线（核心：链接 → 自动音频 → 低成本可信结果）

这是把你"倾向的下一步"具体化成可执行的状态机。单集进来后按顺序走，命中即停：

```
[输入：问题 or 链接]
   │
   ├─ 是问题 → 检索小宇宙 → 选出候选单集（播放量/影响力/新鲜度排序）
   │
   ▼ 对每个入选单集：
   │
   1. 查 transcript 缓存（按 episode_id）
   │     命中 → 直接用全文，0 成本 0 等待  ✅
   │
   2. 查是否有用户粘贴文稿 / 小宇宙官方自动文稿
   │     有 → 入缓存 → 直接用  ✅
   │
   3. 抓公开页面（简介 / Show Notes / 时间轴 / 评论 / 联网资料）
   │     → 立即产出「快速专题 v1」，标注"基于公开资料整理"  ⏱ 1-3 min
   │     → 用户马上能读到第一版（这一步永远先出）
   │
   4. 估算音频时长 → 估算 ASR 消耗
   │     额度足够 + 单集重要度高 → 进入片段/全文转写队列
   │     额度紧张 → 只转写最关键 1-2 期，其余停在 v1
   │
   5. 后台异步转写 → 回填 transcript 缓存 → 自动升级专题为 v2（有全文证据）
```

关键设计原则：

- **第 3 步永远先跑**，保证用户 2-3 分钟内看到 v1，绝不让人空等全文。
- **转写是后台增量**，不阻塞阅读。v1 → v2 是"页面自动长出更多证据"，不是重新生成。
- **同一集永远只转一次**（靠缓存），转写成本随使用次数摊薄到趋近 0。

## 2. 逐个回答你的 10 个问题

### Q1. 只复制链接，怎样低成本拿到高质量内容？
不要把"高质量"等同于"全文"。低成本高质量的组合拳：
1. 先吃**已有 transcript 缓存**（很多热门集别人/你自己转过）。
2. 吃**官方自动文稿**（见 Q2）。
3. 公开页面 + 联网资料做骨架，**只对最关键 1-2 期补全文**做证据。
4. 把"播放量高/影响力高"的集子优先转写，长尾集子停在 v1。
高质量的来源是**结构和证据标注**，不是逐字稿堆量。一篇标清楚"哪些是听完的、哪些是推断的"专题，比一篇假装全听完的更可信。

### Q2. 小宇宙 App 自动文稿能否更顺畅导出？
现状痛点是"用户嫌选中复制麻烦"。合规前提下能优化体验，不能绕过授权：
- **降低复制门槛**：助手页面放一个大输入框 + "粘贴文稿"按钮，配 30 秒动图教"长按全选→复制"。把摩擦从"我要去想怎么弄"降到"照着点两下"。
- **剪贴板监听（本地）**：server 跑在本机，可在用户点了"我去复制"后，轮询系统剪贴板，检测到长文本自动填入。用户只需复制，不需回来粘贴。（macOS 上 `pbpaste` 即可，注意只在用户明确触发后短时间监听，用完即停，别长期读剪贴板。）
- **不要做**：自动化脚本去 App 里抓文稿、逆向接口。这越权且不稳定，也是你不想碰的红线。

### Q3. "链接 → 自动音频 → 低成本转写"体验能否做？
能，就是第 1 节那条流水线。关键不在转写本身，在**缓存 + 降级 + 异步**三件套。用户体感是"贴完链接就有结果，过一会儿结果变厚了"。

### Q4. 怎么做智能片段转写而不是全文？
两步走：
1. **定位关键段**：用时间轴/章节标题/Show Notes 里的小标题，结合用户的问题做语义匹配，挑出最相关的 1-3 个时间区间。
2. **只切那几段音频转写**：`ffmpeg -ss <start> -t <dur>` 切片 → 只把切片送 ASR。一集 60 分钟可能只转 8 分钟，额度省 80%+。
落到证据上：v1.5（片段转写）比纯 Show Notes 可信得多，又远比全文便宜。这其实是比"全文"更该当默认的深挖档。

### Q5. 怎么估算音频时长、提前提示消耗？
- 多数播客音频是直链 mp3/m4a。最稳的方式：抓到音频 URL 后用 `ffprobe -show_entries format=duration` 读时长（只下文件头，不用下全程）。
- 拿不到就用 `Content-Length` ÷ 码率粗估，或读页面上的时长字段。
- 提示话术要具体：「这集约 58 分钟，全文转写约消耗 1 小时 ASR 额度（剩余 X 小时）。建议只转关键 12 分钟（约 0.2 小时）。[只转关键段] [转全文] [先看公开资料版]」

### Q6. 怎么建 transcript 缓存让同一集不重复转写？
一张表搞定，按 `episode_id` 做主键（小宇宙单集 URL 里能解析出稳定 ID）：

```
transcript_cache:
  episode_id   TEXT  PRIMARY KEY     # 稳定唯一 ID
  source       TEXT                  # asr_full | asr_segment | user_paste | official
  status       TEXT                  # pending | partial | done | failed
  segments     JSON                  # [{start, end, text}]，片段转写也能合并
  full_text    TEXT
  audio_dur    REAL
  updated_at   TIMESTAMP
```

规则：任何来源拿到的文稿都写进这张表；新任务先查表；片段转写按区间合并进 `segments`，下次需要别的区间只补缺口。**缓存是这套产品成本曲线往下走的命根子。**

### Q7. 云端额度用完后怎么无感切本地后台转写？
- 维护一个 `asr_quota` 计数器，每次转写前预扣、失败回滚。
- 额度 > 0：走腾讯云（快）。
- 额度 = 0：自动入**本地 Whisper 后台队列**，不打断用户——v1 已经在手，全文慢慢补。完成后页面静默升级 + 给个"该集全文已就绪"的小通知。
- 关键体验：**切换对用户透明**。唯一变化是"补全文从几分钟变成慢一点"，而不是报错或卡住。本地用 `faster-whisper` + 合适的模型（small/medium 平衡速度），能比原生 whisper 快数倍。

### Q8. 怎么让用户 2-3 分钟看到第一版？
靠第 1 节流水线的**第 3 步必先跑**。再加产品上的两个动作：
- 任务卡片状态做成可见进度：`检索中 → 公开资料整理中 → ✅ 初稿可读 → 补全文中(2/5)`。
- v1 一旦 ready 就推"立即阅读"，别等转写。让用户在读 v1 的时候，后台在补 v2。等待感被阅读填满 = 不焦虑。

### Q9. 没全文时怎么不把观点写成已证实事实？
这是可信度的核心，必须在 **prompt + 渲染**两层都管：
- **生成层**：给模型的指令明确分级——只有 `source ∈ {asr, user_paste, official}` 的内容能写成"节目中提到/主播认为"；只有 Show Notes/简介的，必须写成"据节目简介"；联网补充的写"据公开报道"。**禁止把推断写成转述。**
- **渲染层**：每个段落/金句带一个来源徽标（🎧 听稿 / 📝 简介 / 🌐 联网 / 🤔 推断）。专题顶部一行总标注当前可信度等级。
- **金句图**：只允许从 `source=asr/official/user_paste` 的句子生成可分享金句，杜绝把 AI 推断当主播原话传播出去（这是合规和口碑的双重风险）。

### Q10. 多人用时怎么控成本和排队？
- **全局缓存共享**：A 转过的集，B 直接命中——用户越多，边际成本越低。这是这产品规模化最大的优势，要在架构上一开始就做成"缓存是公共资产"。
- **任务队列 + 配额**：每用户每日免费转写额度（如 30 分钟 ASR），超出排队走本地 Whisper 或提示等待。
- **去重转写**：同一集多人同时请求，合并成一个转写任务，结果广播。
- **冷热分层**：热门集预转写并常驻缓存；长尾集按需、低优先级。

## 3. 建议的对外形态（按钮与文案）

主输入框下方，**只放一个主按钮**：

```
[ 🎧 直接研究 · 推荐 ]      ← 默认，走第 1 节流水线
```

折叠在"更多选项"里（高级用户才点）：

```
[ ⚡ 仅公开资料 · 最快0成本 ]
[ 📝 我已有文稿 · 粘贴 ]
[ 📚 全文深研 · 会消耗额度 ]   ← 点击时强制弹消耗预估确认
```

结果页顶部固定一行可信度条：

```
本专题基于：🎧 2 期听稿 · 📝 1 期文稿 · 🌐 联网补充 · 🤔 部分推断
[查看证据明细]   [补全剩余 3 期全文 →]
```

## 4. 落地优先级（给你和 codex 的施工顺序）

按"投入产出比"排，建议从上往下做：

1. **transcript 缓存表 + episode_id 解析**（Q6）。一切的地基，先做。
2. **流水线第 3 步：公开资料 v1 永远先出**（Q8）。立刻改善"等太久"的核心抱怨。
3. **生成层来源分级 + 渲染层来源徽标**（Q9）。可信度护栏，越早越好，避免口碑事故。
4. **音频时长估算 + 消耗预估确认弹窗**（Q5）。让"花钱"变透明。
5. **片段转写**（Q4）。把默认深挖从"全文"降级成"关键段"，省额度主力。
6. **后台异步转写 + v1→v2 自动升级**（Q3/Q1）。
7. **额度耗尽自动切本地 Whisper 队列**（Q7）。
8. **剪贴板辅助导入文稿**（Q2）。体验糖，靠后。
9. **多人共享缓存 / 队列 / 配额**（Q10）。等真要给更多人用再做。

前 3 项做完，你的四个最痛问题（链接体验、ASR 成本、等待时间、可信度）就已经缓解大半。

---

## 5. 一句话总结给 codex

> 把 5 个模式藏进后台变成一条"查缓存→公开资料先出 v1→关键段转写→后台补全文"的自动降级流水线；对外只留一个"直接研究"按钮；用 episode_id 缓存让转写永不重复；用来源徽标保证不把推断写成事实。先做缓存表、v1 先出、来源标注三件事。
